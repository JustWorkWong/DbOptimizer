## 1. 项目概述

### 1.1 项目定位

DbOptimizer 是一个基于 Microsoft Agent Framework (MAF) 的 AI 驱动数据库性能优化平台，通过多 Agent 协作提供智能化的数据库优化建议。

**核心价值**：
- **展示 MAF 能力**：多 Agent 协作、Workflow 编排、Human-in-the-loop
- **透明化 AI 过程**：实时展示 Agent 工作过程、决策推理链、证据引用
- **实用的优化建议**：结合执行计划、索引分析、配置优化，提供可落地的建议

### 1.2 目标用户

- **开发者**：本地调试时快速分析慢查询
- **DBA**：生产环境性能优化、配置调优
- **技术面试官**：评估候选人对 MAF 的理解和应用能力

### 1.3 技术栈

**后端**：
- .NET 10
- ASP.NET Core Web API
- Microsoft Agent Framework (MAF)
- Aspire（本地编排 PostgreSQL / Redis）
- SSE（实时推送）
- Entity Framework Core

**前端**：
- Vue 3
- Element Plus（UI 组件库）
- Monaco Editor（SQL 编辑器）
- ECharts（图表）

**数据库**：
- PostgreSQL（主存储）
- Redis（缓存 + 会话）

**AI**：
- Azure OpenAI / Anthropic Claude
- Microsoft Agent Framework

**支持的目标数据库**：
- MySQL 5.7+
- PostgreSQL 13+

## 2. 核心功能需求

### 2.1 SQL 层调优

#### 2.1.1 手工 SQL 分析

**功能描述**：
- 用户手动输入 SQL 进行分析
- 多 Agent 协作分析（SQL 解析 + 执行计划 + 索引推荐）
- 实时显示 Agent 工作过程

**输入**：
- SQL 文本
- 目标数据库连接信息

**输出**：
- 问题诊断（全表扫描、索引失效、JOIN 顺序）
- 索引推荐（CREATE INDEX DDL + 预估收益）
- SQL 重写建议
- 置信度 + 原因 + 证据引用

**用户故事**：
```
作为开发者
我想输入一条慢 SQL
以便快速获得优化建议
```

#### 2.1.2 慢 SQL 自动抓取

**功能描述**：
- 自动采集目标数据库的慢查询日志
- 定时任务扫描 MySQL slow log / PostgreSQL pg_stat_statements
- 自动触发分析工作流

**配置项**：
- 采集间隔（默认 5 分钟）
- 慢查询阈值（默认 1 秒）
- 最大采集数量（默认 100 条）

**输出**：
- 慢查询列表（按执行时间排序）
- 每条慢查询的优化建议
- 趋势分析（慢查询数量变化）

**用户故事**：
```
作为 DBA
我想自动监控生产数据库的慢查询
以便及时发现性能问题
```

### 2.2 数据库层调优

#### 2.2.1 配置参数优化

**功能描述**：
- 分析数据库配置参数（innodb_buffer_pool_size、shared_buffers 等）
- 结合服务器资源、数据规模、负载特征生成建议
- 预估配置变更的影响

**输入**：
- 当前配置参数
- 服务器资源信息（CPU、内存、磁盘）
- 数据库规模（数据量、表数量）
- 负载特征（QPS、连接数）

**输出**：
- 配置优化建议（参数名 + 建议值 + 当前值）
- 优化原因（为什么要调整）
- 预估收益（性能提升百分比）
- 风险提示（可能的副作用）

**用户故事**：
```
作为 DBA
我想分析数据库配置是否合理
以便优化整体性能
```

### 2.3 人工审核

#### 2.3.1 审核工作流

**功能描述**：
- 所有 AI 建议必须经过人工审核
- 审核不通过时，回流 Workflow 重新生成
- 支持审核意见反馈

**审核选项**：
- **同意**：标记建议为已批准
- **驳回**：拒绝建议，提供驳回原因
- **调整**：修改建议参数，重新生成

**审核界面**：
- 展示 AI 建议详情
- 展示置信度 + 原因 + 证据链
- 提供审核意见输入框
- 显示历史审核记录

**用户故事**：
```
作为 DBA
我想审核 AI 生成的优化建议
以便确保变更的安全性
```

#### 2.3.2 驳回后重跑

**功能描述**：
- 用户驳回建议时，提供驳回原因
- Workflow 从 Checkpoint 恢复，结合驳回原因重新生成
- 最多迭代 3 次，超过后标记为失败

**驳回原因类型**：
- 索引列选择不合理
- 预估收益过于乐观
- 忽略了业务约束
- 其他（自由文本）

**用户故事**：
```
作为 DBA
我想在驳回建议时提供反馈
以便 AI 生成更符合预期的建议
```

### 2.4 历史记录与版本管理

#### 2.4.1 历史任务查看

**功能描述**：
- 查看所有历史分析任务
- 按时间、状态、数据库筛选
- 查看任务详情（输入、输出、执行过程）

**任务状态**：
- 运行中
- 等待审核
- 已完成
- 已失败

**用户故事**：
```
作为 DBA
我想查看历史优化任务
以便追溯优化效果
```

#### 2.4.2 建议版本演进

**功能描述**：
- 同一个 SQL 的多次分析结果对比
- 展示建议的演进过程（驳回 → 调整 → 再生成）
- 支持版本回滚

**用户故事**：
```
作为 DBA
我想对比同一个 SQL 的不同优化方案
以便选择最优方案
```

### 2.5 运行时间线

**功能描述**：
- 可视化展示 Workflow 执行过程
- 时间线展示每个 Executor 的执行时间
- 展示 Agent 消息、工具调用、决策过程

**时间线元素**：
- Executor 开始/结束时间
- Agent 消息（System / User / Assistant / Tool）
- 工具调用（MCP 调用、数据库查询）
- Checkpoint 保存点
- 人工审核节点

**用户故事**：
```
作为开发者
我想查看 Workflow 的执行过程
以便理解 AI 的决策逻辑
```

## 3. 非功能需求

### 3.1 性能要求

- **SQL 分析响应时间**：< 30 秒（单条 SQL）
- **慢查询采集延迟**：< 5 分钟
- **SSE 推送延迟**：< 500ms
- **并发支持**：10 个并发分析任务

### 3.2 可用性要求

- **系统可用性**：99%（单项目部署，允许维护窗口）
- **错误恢复**：Workflow 失败后可从 Checkpoint 恢复
- **数据持久化**：所有 Agent 消息、决策、审核记录永久保存

### 3.3 安全要求

- **连接字符串加密**：AES-256 加密存储
- **敏感信息过滤**：不记录密码、Token 到日志
- **输入验证**：防止 SQL 注入
- **Token 限制**：单次分析最多消耗 50k tokens

### 3.4 可维护性要求

- **日志记录**：结构化日志（Serilog）
- **监控指标**：Workflow 执行时间、Token 消耗、错误率
- **代码覆盖率**：> 80%

## 4. 约束条件

### 4.1 第一版范围

**必须实现**：
- 单项目模式运行（不做多租户）
- Aspire 编排 API / AgentRuntime / Web / PostgreSQL / Redis
- MySQL & PostgreSQL MCP 同时接入
- 数据库层调优工作流
- SQL 调优工作流（手工输入 + 慢 SQL 自动抓取）
- 人工审核 + 审核驳回后回流重跑
- Agent 全量持久化 + Workflow checkpoint
- 置信度 + 原因 + 证据链展示
- 历史任务与建议版本查看

**第一版不做**：
- 用户/租户体系
- 自动执行变更（参数/索引/SQL rewrite）
- 平台自身对外暴露 MCP Server
- 复杂 RAG / 向量检索（Phase 2）
- 多数据库类型扩展（只做 MySQL / PostgreSQL）

### 4.2 技术约束

- **MAF 版本**：使用最新稳定版
- **Aspire 用途**：仅用于本地编排，不用于生产部署
- **MCP 集成**：直接接入现有 MySQL / PostgreSQL MCP，不自己实现
- **前端框架**：Vue 3（不使用 Blazor）
- **实时推送**：SSE（不使用 SignalR）

## 6. 数据需求

### 6.1 核心实体

- **Workflow Session**：工作流会话
- **Workflow Checkpoint**：工作流检查点
- **Executor Execution**：执行器执行记录
- **Agent Message**：Agent 消息
- **Target Database**：目标数据库
- **Slow Query**：慢查询
- **Index Recommendation**：索引推荐
- **Configuration Recommendation**：配置推荐

### 6.2 数据保留策略

- **Workflow Session**：保留 90 天
- **Executor Execution**：保留 30 天
- **Agent Message**：保留 30 天
- **Slow Query**：保留 30 天
- **Recommendation**：永久保留

## 10. 风险与依赖

### 10.1 技术风险

- **MAF 稳定性**：MAF 是新框架，可能存在 Bug
  - **缓解措施**：准备降级方案（自己实现状态机）

- **MCP 可用性**：MySQL / PostgreSQL MCP 可能不稳定
  - **缓解措施**：准备 fallback（直接用 ADO.NET / Npgsql）

- **Token 成本**：多 Agent 协作 Token 消耗大
  - **缓解措施**：启用 Prompt Caching，限制单次分析 Token 上限

### 10.2 外部依赖

- **Azure OpenAI / Anthropic API**：需要稳定的 API 访问
- **MySQL / PostgreSQL MCP**：依赖社区维护的 MCP Server
- **.NET 10**：需要最新的 .NET SDK

## 11. 术语表

| 术语 | 定义 |
|------|------|
| MAF | Microsoft Agent Framework，微软的 Agent 框架 |
| Workflow | 工作流，由多个 Executor 组成 |
| Executor | 执行器，Workflow 中的一个执行单元 |
| Checkpoint | 检查点，Workflow 的状态快照 |
| MCP | Model Context Protocol，模型上下文协议 |
| SSE | Server-Sent Events，服务器推送事件 |
| Aspire | .NET Aspire，用于编排云原生应用 |
| Human-in-the-loop | 人工介入循环，AI 决策需要人工审核 |
