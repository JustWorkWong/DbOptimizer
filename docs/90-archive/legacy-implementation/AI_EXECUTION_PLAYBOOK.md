# DbOptimizer AI 执行手册

**目的**: 规范后续 AI / Agent 如何消费实施计划，避免上下文失控、任务边界发散、交接信息丢失。  
**适用对象**: Codex、Claude Code、其他能按 checklist 执行任务的编码 Agent。

---

## 1. 执行原则

1. 一次只做一个任务
2. 先读任务卡，不先读全仓库
3. 先做最小改动，不做额外重构
4. 每次完成必须写摘要
5. 没验证到的内容必须明确说明

---

## 2. 上下文控制规则

## 2.1 每次只加载必要文件

建议每个任务只优先读取：

- 任务卡中列出的文件
- 与当前任务直接相关的 1-2 个调用链文件

不要一开始就加载：

- 全部 docs
- 全部前端页面
- 全部工作流执行器

## 2.2 超过 5 个文件时必须做中间摘要

摘要格式至少包含：

- 当前理解的目标
- 已确认的关键实现点
- 已读取的文件
- 下一步准备修改的文件

## 2.3 长任务要分段交接

如果一个任务无法在单轮内稳定完成，必须产出阶段摘要：

- 做完了什么
- 还差什么
- 哪些点未验证
- 继续执行时先看什么

---

## 3. 任务执行模板

每次执行任务前，先输出下面这四点：

1. 当前任务 ID
2. 当前目标
3. 计划修改文件
4. 计划验证方式

示例：

```markdown
当前任务: TASK-A1
目标: 统一工作流结果模型
计划修改文件:
- src/DbOptimizer.API/Api/WorkflowApi.cs
- src/DbOptimizer.API/Api/DashboardAndHistoryApi.cs
- src/DbOptimizer.API/Api/ReviewApi.cs
计划验证:
- 查询 workflow 接口
- 查询 history 接口
- 查询 review 接口
```

---

## 4. 摘要与交接规则

## 4.1 每次结束必须写“完成摘要”

模板：

```markdown
任务完成摘要:
- 任务ID:
- 已修改文件:
- 已实现功能:
- 已执行验证:
- 验证结果:
- 未完成项:
- 推荐下一任务:
```

## 4.2 每个摘要都必须足够让下一个 Agent 接手

至少要回答：

- 现在代码处于什么状态
- 哪些文件刚改过
- 下一个 Agent 不应该重复做什么
- 还有哪些风险或待验证点

---

## 5. AI 在本项目里的技术边界

## 5.1 当前阶段默认不要做的事

- 不主动引入 Microsoft Agent Framework
- 不主动接入 Azure OpenAI / Claude SDK
- 不主动把前端重构成 Router + Pinia + Element Plus 全量架构
- 不主动做大规模目录重构

## 5.2 当前阶段应该优先做的事

- 打通配置调优闭环
- 打通慢 SQL 自动分析闭环
- 做规则版 SQL Rewrite
- 保证 history / review / workflow status 正常

---

## 6. Agent 特别注意事项

## 6.1 注意结果模型

- 凡是涉及 workflow/history/review 的任务
- 都必须先确认当前结果类型是否仍被写死为 `OptimizationReport`
- 这是本项目当前最容易踩坑的地方

## 6.2 注意前端单页现状

- 当前前端实际是单页 `App.vue`
- 文档中的 Router/Pinia 结构不是当前事实
- 不要按文档想当然去找不存在的文件

## 6.3 注意任务边界

- 当前任务若是 `TASK-A4`
- 就不要顺手把 `TASK-C3` 的 dashboard 重构一起做了

## 6.4 注意验证说明

- 如果因为环境、构建、依赖原因没法跑验证
- 必须写清楚没跑什么、原因是什么

---

## 7. 推荐任务节奏

推荐 AI 节奏：

1. 读 `IMPLEMENTATION_INDEX.md`
2. 只读当前任务卡
3. 读取必要代码
4. 修改
5. 最小验证
6. 输出摘要
7. 再接下一个任务

不推荐：

1. 一次性读取全部 docs
2. 一次性做完整个 Sprint
3. 一次性前后端大改

---

## 8. 推荐交接摘要模板

后续每次任务结束建议输出：

```markdown
阶段摘要:
- 当前任务:
- 当前状态:
- 已改文件:
- 关键实现点:
- 已验证:
- 未验证:
- 下一步建议:
```

---

## 9. 面向长链任务的“压缩摘要”要求

如果任务执行时间较长，或上下文已经很大，摘要必须特别短，只保留：

1. 目标
2. 已改文件
3. 结果
4. 风险
5. 下一步

示例：

```markdown
压缩摘要:
- 目标: 打通配置调优结果模型
- 已改: WorkflowApi.cs, ReviewApi.cs
- 结果: workflow/review 已能区分 SQL 与 Config 结果
- 风险: history 接口还未同步
- 下一步: 执行 TASK-A6
```

---

## 10. 任务完成定义

AI 只有在同时满足以下条件时，才应认为任务完成：

1. 代码已修改
2. 任务目标已覆盖
3. 做过最小验证
4. 写了完成摘要
5. 明确指出下一步建议
